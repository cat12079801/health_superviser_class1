# 進捗同期（ログイン）機能 要件定義書

## 1. 概要

複数端末間で学習進捗を同期できるようにする。ログイン機能を導入し、ログインしたユーザの進捗を外部サービスに保存する。ログインは任意とし、未ログインでも従来どおり `localStorage` のみで全機能を利用できる。

本機能は [REQUIREMENTS.md](../REQUIREMENTS.md) 9章でスコープ外としていた「端末間の進捗同期」を、バックエンドを持つ構成へ拡張するものである。ただし配信は引き続き GitHub Pages（静的配信）を維持し、サーバコードは持たない。

## 2. 決定事項

要件確定にあたり、以下を決定した。

| 項目 | 決定 |
|------|------|
| 認証・保存基盤 | Supabase（無料枠で開始） |
| ログイン手段 | Google ログインのみ |
| 同期・競合解決 | 問題単位マージ（各回答に `updatedAt` を持たせ、問題 `id` ごとに新しい方を採用） |
| 未ログイン利用 | 従来どおり利用可。ログイン時に既存の localStorage 進捗をアカウントへ初回マージする |
| 既存 localStorage 進捗の時刻欠損 | 十分古い時刻で補完する |
| 保存形式 | 1回答1行（`user_id × question_id`） |
| 運用方針 | 無料枠の自動停止を許容する（停止してもデータは保持されるため、利用時に復帰操作を行う） |

## 3. 基盤（Supabase）

Supabase はオープンソースの BaaS であり、認証・PostgreSQL・自動生成 API を自前サーバなしで利用できる。本機能では次を用いる。

- **Auth**: Google OAuth ログイン。
- **Database（PostgreSQL）**: 進捗データの保存先。
- **JS SDK**: 静的サイトのフロントから直接読み書きする（サーバコード不要）。
- **RLS（行レベルセキュリティ）**: 「自分の `user_id` の行のみ読み書き可」を DB 側で強制する。全ユーザの進捗が同一テーブルに格納されるため、他人の行を保護する目的で必須とする。

### 3.1 コストと運用

- 無料枠（DB 500MB / 認証 月間アクティブ 5万人 / 帯域 5GB）で開始する。本用途のデータ量では容量に十分な余裕がある。
- 無料枠では 7 日間アクセスがないとプロジェクトが一時停止する。**一時停止でもデータは消えず保持される**。停止した場合は管理画面から復帰操作を行う。
- 個人学習用途では停止を許容する方針とする。keep-alive や Pro プラン（月 $25）への移行は、運用して不満が生じた時点で再検討する。

## 4. 同期の仕組み（問題単位マージ）

### 4.1 前提となるスキーマ拡張

問題単位で競合を判定するため、各回答に `updatedAt`（その問題を最後に更新した時刻）を持たせる。現行スキーマは状態全体に 1 つの `updatedAt` しか持たず、端末ごとの「どの問題をいつ更新したか」を区別できないためである。

### 4.2 マージアルゴリズム

2 つの `answers`（ローカルとリモート）を統合する際、問題 `id` をキーに次のとおり突き合わせる。

```
merged = {}
すべての id（local の id ∪ remote の id）について:
    片方にしか存在しない  → その 1 件を採用
    両方に存在する        → updatedAt が新しい方を採用
```

競合判定を問題 1 問ごとに独立して行うため、「端末 A で問 1、端末 B で問 2 を解いた」場合でも、どちらの更新も失われず両方が残る。状態全体を新しい方で上書きする Last Write Wins（LWW）と異なり、片方の端末の作業が丸ごと消える事故を防げる。

### 4.3 `attempts`（挑戦回数）の扱い

`lastChoice` / `correct` は「新しい方を採用」で確定する。`attempts` は本来端末間で合算すべき性質だが、同期のたびに二重計上する危険があり実装が複雑になる。学習アプリにおける `attempts` の厳密さは重要度が低いため、**正確さより単純さを優先し「新しい方を採用」とする**。この割り切りを許容する。

### 4.4 同期のタイミング

1. **ログイン直後（初回マージ）**: リモートを取得し、ローカルとマージした結果を localStorage と Supabase の両方へ書き戻す。
2. **回答ごと**: ログイン中は、回答した 1 問を `updatedAt` 付きで Supabase へ upsert する。
3. **起動時・フォアグラウンド復帰時**: リモートを取得して再マージする。
4. **オフライン時**: 通信に失敗しても localStorage で継続し、復帰時に同期する。

## 5. 機能要件

1. **ログイン / ログアウト**: Google アカウントでログインする。ログイン状態と本人（メール／名前）を表示し、ログアウト操作を提供する。
2. **未ログイン利用**: 未ログインでも従来どおり localStorage のみで全機能が動作する。
3. **初回マージ**: ログイン成功時に localStorage の進捗とリモートを問題単位でマージし、両方へ書き戻す。
4. **回答の同期書き込み**: ログイン中は回答ごとにリモートへ upsert する。
5. **再マージ**: 起動時・復帰時にリモートを取得して再マージする。
6. **オフライン耐性**: 通信失敗時も学習を継続でき、復帰時に同期する。
7. **進捗リセット**: 既存のリセット操作は、ログイン中はリモートの該当ユーザ行も削除する。

## 6. データ設計

### 6.1 localStorage（既存の拡張）

キーは既存の `eisei1.progress.v1` を継続する。各 `answers[id]` に `updatedAt` を追加する。

```json
{
  "answers": {
    "law-haz-001": {
      "lastChoice": 1,
      "correct": false,
      "attempts": 3,
      "updatedAt": "2026-07-01T09:00:00Z"
    }
  },
  "updatedAt": "2026-07-01T09:00:00Z"
}
```

既存データには問題単位の `updatedAt` が存在しないため、マイグレーション時に十分古い時刻（例: `1970-01-01T00:00:00Z`）で補完する。これによりリモート側に新しい回答があればそちらが優先され、既存データがマージで不当に勝つことを防ぐ。

### 6.2 Supabase（1回答1行）

`progress` テーブルを設け、1 行を「あるユーザのある問題への回答」とする。

| カラム | 内容 |
|--------|------|
| `user_id` | 認証ユーザの ID（`auth.users` と連携） |
| `question_id` | 問題 `id` |
| `last_choice` | 最後に選んだ選択肢インデックス |
| `correct` | 正誤 |
| `attempts` | 挑戦回数 |
| `updated_at` | その問題を最後に更新した時刻 |

- 主キーは `(user_id, question_id)` とし、回答ごとの upsert を可能にする。
- **RLS を有効化し、`user_id = auth.uid()` の行のみ select / insert / update / delete を許可する**（必須）。

## 7. 非機能・運用要件

- 配信は GitHub Pages を維持する。
- Supabase の anon key はフロントに露出する前提とし（公開しても問題ない設計）、保護は RLS で担保する。
- Google OAuth のリダイレクト許可 URL に GitHub Pages のドメインを登録する。
- 個人の学習進捗という個人データを外部保存する点を、プライバシーに関する記載へ追記する。

## 8. 実装段階（提案）

1. Supabase プロジェクト作成、Google OAuth 設定、`progress` テーブルと RLS ポリシー定義。
2. localStorage スキーマへの `updatedAt` 追加と、既存データの時刻補完マイグレーション。
3. マージ関数の実装と単体テスト（純ロジックのため `scripts` 配下でテスト可能）。
4. ログイン UI の追加と、`store.js` の同期対応（回答時 upsert・起動時再マージ）。
5. オフライン耐性と再同期の調整。

## 9. スコープ外

- Google 以外のログイン手段（メール＋パスワード、マジックリンク等）。
- keep-alive による自動停止回避および Pro プランでの常時稼働。
- 進捗以外のデータ（設定・お気に入り等）の同期。
