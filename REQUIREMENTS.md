# 第一種衛生管理者試験 学習アプリ 要件定義書

## 1. 概要

第一種衛生管理者試験の学習を行う Web アプリである。関係法令・労働衛生・労働生理の**全5カテゴリ**を対象とする。
1問ごとに回答・解説を確認する一問一答形式とする。問題データと画面は静的ファイルのみで完結させる。ログイン時のみ端末間で進捗を同期する任意機能を備える（2 章・5.3 節を参照）。

## 2. 対象・前提

- 利用端末: タブレット（iPad）、iPhone、Android。スマホ縦持ちを基本とし、レスポンシブに対応する。Android/Chrome を含む端末でホーム画面へインストールできるよう、PWA（Web App Manifest・Service Worker）に対応する。
- 学習状態は端末ごとに `localStorage` で管理する（cookie は容量・送信の観点で不採用）。
- ログインは任意機能とする。既定ではログイン機構を用いず `localStorage` のみで完結する。Google ログイン時に限り、進捗を Supabase に保存して端末間で同期する（5.3 節）。
- 問題データは Claude が追加・更新できる形式（JSON）で管理する。

## 3. 試験形式の調査結果（設計の根拠）

第一種衛生管理者試験の公開情報に基づく。

- 出題形式: **五肢択一式**。設問は「正しいものはどれか」「誤っているものはどれか」が中心。
  選択肢が組合せ（A〜Dのうち正しいものの組合せ）になる出題も存在する。
- 関係法令の構成:
  - 関係法令（有害業務に係るもの）: 10問 / 80点
  - 関係法令（有害業務以外のもの）: 7問 / 70点
- 合格基準: 各科目40%以上（足切り）かつ全科目合計60%以上。

→ 本アプリは上記の **五肢択一（単一回答）** を標準形式とし、組合せ問題も同一の選択肢構造で表現できるようにする。
   正誤（○×）形式は試験本体に存在しないため対象外とする。

## 4. 機能要件

### 4.1 出題・回答

- 1問ずつ表示し、5つの選択肢から1つを選択して回答する。
- 回答後に **正誤判定** と **解説** を表示する。
- 「次の問題へ」で次問に進む。
- 出題順は出題モード（4.3）に従う。

### 4.2 学習記録（端末ごと）

- 各問題の **正誤（正解 / 不正解）** を記録する。
- 各問題の **回答済み / 未回答** 状態を記録する。
- 全体・カテゴリ別の **正答率** を表示する。
- **進捗リセット** ボタンで記録を消去できる。

### 4.3 出題モード

- **全問出題**: 全カテゴリの問題を対象に出題する。
- **カテゴリ別出題**: 「法令（有害）」「法令（一般）」「衛生（有害）」「衛生（一般）」「労働生理」の分野で絞り込んで出題する。
- **復習モード**: 不正解の問題・未回答の問題のみを抽出して再出題する。

### 4.4 問題管理

- 問題は JSON ファイルで管理し、Claude が直接編集して追加・更新できる。
- スキーマ（後述）に沿って記述する。

## 5. データ設計

### 5.1 問題データ（静的 JSON）

カテゴリごとのファイル（1ファイル＝1カテゴリ。`data/law_hazardous.json` 等）に分割し、`data/categories.json` のマニフェスト（各カテゴリの `label` / `prefix` / `file` / `implemented`）に列挙する。各ファイルの構造は次のとおりとする。命名規則と全5カテゴリの詳細は [docs/categories.md](docs/categories.md)、スキーマの詳細は [README.md](README.md) に定める。

```json
{
  "version": "1",
  "questions": [
    {
      "id": "law-haz-001",
      "category": "law_hazardous",
      "tags": ["安衛法", "作業主任者"],
      "question": "次の作業のうち、作業主任者の選任が義務付けられていないものはどれか。",
      "choices": [
        "高圧室内作業",
        "..."
      ],
      "answerIndex": 2,
      "difficulty": "standard",
      "explanation": "...解説本文...",
      "reference": "安衛法第14条 / 安衛令第6条"
    }
  ]
}
```

- `category`: `law_hazardous` / `law_general` / `hygiene_hazardous` / `hygiene_general` / `physiology` の5カテゴリ。定義は `data/categories.json`、命名規則は [docs/categories.md](docs/categories.md) に定める。
- `answerIndex`: 正解の選択肢インデックス（0始まり）。組合せ問題も選択肢文に組合せを書くことで同一構造で表現する。
- `difficulty`: 難易度を表す必須フィールド。`hard` / `standard` / `easy` のいずれかとする（方針は CLAUDE.md を参照）。
- `tags`: 法令名など。将来の細分類・検索に利用する。
- `reference`: 根拠条文（任意）。
- `columns`: 用語を補足するコラム（任意）。記述方針は CLAUDE.md に定める。

### 5.2 学習状態（localStorage）

キー: `eisei1.progress.v1`

```json
{
  "answers": {
    "law-haz-001": { "lastChoice": 1, "correct": false, "attempts": 3, "updatedAt": "2026-06-23T00:00:00Z" }
  },
  "updatedAt": "2026-06-23T00:00:00Z"
}
```

- 問題 `id` を主キーに正誤・回答回数を保持する。
- 各回答は `updatedAt`（ISO 8601 UTC）を持つ。同期時に問題単位で新しい方を採用してマージするためである。旧スキーマの欠損値は十分古い時刻で補完する。
- 問題データの増減に対し、未知の `id` は未回答として扱い、削除済み `id` は無視する（疎結合）。

### 5.3 進捗同期（任意・Supabase）

Google ログイン時のみ、進捗を Supabase の `progress` テーブルへ保存し、端末間で同期する。未ログイン・未設定・通信失敗のいずれでもアプリは `localStorage` のみで動作を継続する（同期は任意機能でありベストエフォートとする）。

- `progress` テーブルの列は `user_id` / `question_id` / `last_choice` / `correct` / `attempts` / `updated_at` とする。行単位アクセスは Supabase の RLS で `user_id` により制御する。
- 同期方向: ログイン・フォアグラウンド復帰・オンライン復帰時にリモートとローカルを問題単位でマージ（`updated_at` の新しい方を採用）し、双方へ書き戻す。
- 接続情報（URL・publishable key）は公開してよい値とし、保護は DB 側の RLS に委ねる。

## 6. 画面構成

1. **トップ / モード選択**: 全問・カテゴリ別・復習を選択する。全体正答率と進捗を表示する。
2. **出題画面**: 設問・選択肢・回答ボタンを表示し、回答後に正誤・解説・「次へ」を表示する。あわせて進捗（n/N）を表示する。
3. **結果 / 統計画面**: カテゴリ別正答率を表示し、その下に難易度（難 / 並 / 易）別の正答率を細分化して表示する。あわせてリセットボタンを表示する。

## 7. 技術構成

- **素の HTML / CSS / JavaScript（ビルド工程なし）と JSON データを採用する。**
- フレームワーク・バンドラは使わない（問題追加の手軽さと静的ホスティング適合性を優先する）。
- モジュールは ES Modules（`<script type="module">`）で分割する。
- 想定構成:

```
/
├── index.html
├── manifest.webmanifest      # PWA マニフェスト
├── sw.js                     # Service Worker（オフライン対応・PWA インストール要件）
├── icons/                    # PWA アイコン（192 / 512 / maskable / apple-touch）
├── css/style.css
├── js/
│   ├── app.js        # 画面制御・Service Worker 登録
│   ├── store.js      # localStorage 読み書き・進捗マージ
│   ├── quiz.js       # 出題ロジック・正誤判定・問題データ読み込み
│   ├── sync.js       # Supabase 認証・進捗同期（任意機能）
│   └── config.js     # Supabase 接続情報
└── data/
    ├── categories.json        # カテゴリ定義（読み込むファイルのマニフェスト）
    ├── law_hazardous.json     # 関係法令（有害業務に係るもの）
    ├── law_general.json       # 関係法令（有害業務以外のもの）
    ├── hygiene_hazardous.json # 労働衛生（有害業務に係るもの）
    ├── hygiene_general.json   # 労働衛生（有害業務以外のもの）
    └── physiology.json        # 労働生理
```

## 8. デプロイ先の比較（メリット・デメリット）

| 候補 | メリット | デメリット |
|------|----------|------------|
| **GitHub Pages** | 無料。Git push のみで公開。リポジトリと一体で管理が簡単。本用途に最も素直。 | 独自ビルドパイプラインは弱い。プライベートリポジトリだと有料プラン要件あり。 |
| **Cloudflare Pages** | 無料枠が広い。CDN が高速。カスタムドメイン・HTTPS 容易。 | GitHub 連携・アカウント設定がやや手間。今回の小規模用途には機能過剰。 |
| **Vercel** | デプロイ体験が洗練。プレビュー環境が自動生成。 | フレームワーク前提の機能が多く、素の静的サイトには過剰。商用利用は規約注意。 |

**推奨: GitHub Pages。** 静的ファイルのみ・小規模・個人学習用途であり、リポジトリへの push だけで公開・更新が完結する点が要件に最も合致する。CDN 速度や独自ドメインを重視するなら Cloudflare Pages を次点とする。
